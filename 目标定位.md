# Kupola 架构

> **目标定位**：零外部框架依赖的声明式 UI 组件系统  
> **对标生态位**：HTMX（数据获取）+ Alpine.js（声明式交互）+ Bootstrap（组件丰富度）  
> **核心原则**：聚焦组件本身，给 **不想引入 Vue/React 的项目** 提供开箱即用的交互组件 + 完整数据系统


## Kupola 战略

### 一、现状总结

Kupola 目前有 **三层架构**：

| 层级 | 内容 | 文件 |
|------|------|------|
| **CSS 设计系统** | 13 个 CSS 文件，组件样式 + 主题 + 响应式 | `css/` |
| **JS 组件层** | 30+ UI 组件（dropdown, modal, table...） | `js/*.js` |
| **核心基础设施** | 组件基类 + 响应式数据 + 声明式依赖 + 生命周期 | `component.js`, `data-bind.js`, `depends.js`, `registry.js`, `initializer.js`, `kupola-core.js` |

> 注：Phase 1 清理已删除 `composition-api.js`、`mount.js` 等 16 个文件（约 5,275 行），移除重复 Vue 能力的代码，聚焦组件本身。


### 二、战略建议：聚焦核心差异化

**Kupola 的真正价值在于：零外部框架依赖的声明式 UI 组件系统 + 完整数据层。**

不是 Vue 的替代品，而是给 **不想引入 Vue/React 的项目** 提供开箱即用的交互组件和数据能力。后端只需输出标准 HTML，前端一行 CDN 引入即获得完整交互。


### 三、核心差异化优势

| 能力 | Kupola | Alpine.js / Stimulus | HTMX |
|------|--------|---------------------|------|
| **开箱即用组件** | 30+ 个 | 0 个（用户自己写） | 0 个 |
| **声明式数据获取** | `useDeps()` 多数据源 + 缓存 | 无 | HTML 属性驱动 |
| **响应式系统** | `ref` / `store` / 完整 reactive | 极简响应式 | 无 |
| **CSS 设计系统** | 完整主题 + 品牌色 + 响应式 | 无 | 无 |
| **体积 (gzip)** | ~88 KB（含全部组件） | ~10-15 KB | ~15 KB |

**Kupola 的独特定位**：同时拥有 HTMX 的数据获取能力（`useDeps` 声明式依赖）和 Alpine.js 的声明式交互，外加 30+ 生产级 UI 组件——这是三者中唯一同时具备的。


### 四、目标场景

**核心场景**：任何能输出 HTML 或提供 REST API 的 Web 应用——后端语言与架构无关

#### 4.1 模板渲染模式（SSR）

后端用模板引擎渲染完整 HTML，Kupola 在浏览器端激活交互：

| 后端技术栈 | 模板引擎 | 集成方式 |
|-----------|---------|----------|
| Python (Flask / Django / FastAPI) | Jinja2 / Django Templates | 模板直出 |
| Go (Gin / Echo / Fiber) | html/template | 模板直出 |
| Rust (Actix-web / Axum) | Tera / Askama | 模板直出 |
| Ruby on Rails | ERB / ViewComponent | 模板直出 |
| Java (Spring Boot) | Thymeleaf / JSP | 模板直出 |
| ASP.NET | Razor Pages / MVC Views | 模板直出 |
| PHP (Laravel) | Blade Templates | 模板直出 |
| Node.js (Express / Koa) | EJS / Pug / Handlebars | 模板直出 |

#### 4.2 RESTful API 模式

后端只提供 JSON API，前端用静态 HTML + `useDeps()` 获取数据：

```html
<!-- 静态 HTML 骨架 -->
<div data-dropdown data-deps-source="/api/views">
  <button class="ds-dropdown__trigger"><span>加载中...</span></button>
  <ul class="ds-dropdown__menu"></ul>
</div>

<script type="module">
  import { useDeps } from 'kupola';
  
  // 从 REST API 获取数据，自动填充组件
  const { views } = useDeps({}, {
    views: { source: '/api/views', staleTime: 60000 }
  });
</script>
```

**RESTful 后端示例**（任何语言，返回 JSON 即可）：
```json
// GET /api/views
{ "views": [{ "id": "1", "name": "概览" }, { "id": "2", "name": "详情" }] }
```

#### 4.3 混合模式（推荐）

服务端渲染 HTML 骨架 + 客户端通过 `useDeps()` 获取动态数据：
- 首屏由 SSR 快速渲染（SEO 友好）
- 动态数据由 `useDeps()` 异步获取（交互丰富）

#### 4.4 模式对比

| 模式 | 数据来源 | 首屏速度 | SEO | 适用场景 |
|------|---------|---------|-----|----------|
| **模板渲染** | 服务端嵌入 HTML | 快 | 好 | 内容型页面 |
| **REST API** | 客户端 `useDeps()` | 慢 | 差 | 动态数据密集型 |
| **混合** | SSR 骨架 + API 数据 | 中 | 中 | 推荐方案 |

**为什么都能用**：
- Kupola 是纯前端库，只要求浏览器环境
- 模板模式：组件通过 HTML 属性声明（`data-dropdown`），自动初始化
- API 模式：`useDeps()` 对接任何 REST API，自动缓存/重试/刷新
- 无需 Node.js 构建流程，CDN 引入即可

**典型使用模式**：
- 传统 Web 应用渐进增强（已有模板，加交互）
- 遗留系统现代化（不换后端，升级前端体验）
- 轻量级静态站点 / 文档站
- 嵌入式 Widget / 第三方集成
- SPA 风格的 SSR 应用（混合模式）


### 五、交互机制：后端零侵入

后端完全不需要知道 Kupola 的存在。交互通过三层自动激活：

```
后端输出 HTML                    Kupola JS 自动处理
─────────────                   ──────────────────

  <div data-dropdown>     →     initializer 扫描 [data-dropdown]
    <button class="ds-dropdown__trigger">  →  找到 trigger
    <ul class="ds-dropdown__menu">         →  找到 menu
      <li data-value="a">选项A</li>        →  绑定 item click
    </ul>
  </div>                  →     initDropdown() 完成，监听 click/hover/keyboard
```

> **澄清**：「后端输出 HTML」不是指在代码里拼 HTML 字符串。开发者写的是**模板文件**（本质是 HTML + 动态占位符），后端框架的模板引擎自动渲染成 HTML 发给浏览器。模板文件就是正常的 HTML 页面，只是多了几个动态占位符（`{{ }}`、`{% %}`、`th:` 等）。

**模板文件示例**（Django `templates/dashboard.html`）：
```html
{% extends "base.html" %}
{% block content %}
<div class="page-header">
  <h1>{{ page_title }}</h1>
  <div data-dropdown>
    <button class="ds-dropdown__trigger">
      <span>选择视图</span>
    </button>
    <ul class="ds-dropdown__menu">
      {% for view in available_views %}
      <li data-value="{{ view.id }}">{{ view.name }}</li>
      {% endfor %}
    </ul>
  </div>
</div>
{% endblock %}
```

开发者写的就是 HTML 文件，用模板语法插入动态数据。后端框架渲染后发给浏览器，Kupola 的 JS 在浏览器端扫描 `data-*` 属性，自动激活交互。

**第 1 层：声明式发现**（`initializer.js`）

模板中的 HTML 属性，Kupola 自动识别并初始化：
```html
<div data-dropdown>...</div>
<div data-modal id="confirm-dialog">...</div>
<div data-datepicker data-format="yyyy-MM-dd">...</div>
```

`kupolaInitializer` 维护 20+ 个组件的选择器映射（`data-dropdown` → `initDropdown`），扫描 DOM 后自动调用。

**第 2 层：组件实例化 + 事件绑定**

```
initDropdown(element)
  → new Dropdown(element, options)    // 读取 data-* 属性作为配置
    → .init()
      → 绑定 trigger click / hover
      → 绑定 menu item click
      → 绑定 document click（点击外部关闭）
      → 绑定 keyboard（↑↓ Enter Esc）
```

**第 3 层：动态元素自动发现**（MutationObserver）

`kupolaBootstrap()` 启动 MutationObserver，后续通过 JS/AJAX 动态插入的 `data-*` 元素也会被自动初始化，无需手动调用。

**各后端框架模板等价写法**：

| 后端框架 | 模板文件 | 动态语法 |
|---------|----------|----------|
| Django | `templates/*.html` | `{% for %}` / `{{ var }}` |
| Go Gin | `templates/*.html` | `{{range .Items}}` / `{{.Field}}` |
| Rails | `app/views/*.html.erb` | `<% @items.each %>` / `<%= item.name %>` |
| ASP.NET | `Pages/*.cshtml` | `@foreach` / `@Model.Field` |
| Spring | `templates/*.html` | `th:each` / `${item.name}` |
| Laravel | `resources/views/*.blade.php` | `@foreach` / `{{ $item->name }}` |

前端只需引入一行：
```html
<script src="https://cdn.jsdelivr.net/npm/@kupola/kupola/dist/kupola.esm.js" type="module"></script>
```

这就是 **"后端零侵入"** 设计——后端写正常模板，前端声明式激活交互。


### 六、数据适配：后端 JSON 格式转换

实际开发中后端返回的 JSON 格式往往和前端组件期望的格式不一致，需要转换。

#### 6.1 `useDeps()` 的 `transform` 配置（推荐）

在数据获取阶段直接转换：

```js
const { users } = useDeps({}, {
  users: {
    source: '/api/users',
    // 后端返回: { data: { list: [...], total: 100 } }
    // 组件期望: [{ id, name, email }]
    transform: (response) => {
      return response.data.list.map(user => ({
        id: user.user_id,
        name: `${user.first_name} ${user.last_name}`,
        email: user.contact.email
      }));
    }
  }
});
```

#### 6.2 订阅后处理

数据获取后，在订阅回调中转换：

```js
const { orders } = useDeps({}, {
  orders: '/api/orders'
});

// 后端返回: { orders: [{ order_id, customer_name, items: [...] }] }
orders.subscribe(data => {
  const formatted = data.orders.map(order => ({
    id: order.order_id,
    customer: order.customer_name,
    itemCount: order.items.length,
    total: order.items.reduce((sum, item) => sum + item.price, 0)
  }));
  
  // 更新到 ref 或直接操作 DOM
  orderListRef.value = formatted;
});
```

#### 6.3 中间层封装（多处复用）

对于多处复用的转换逻辑，封装工具函数：

```js
// utils/data-transform.js
export function normalizeUser(apiResponse) {
  return {
    id: apiResponse.user_id,
    displayName: `${apiResponse.first_name} ${apiResponse.last_name}`,
    avatar: apiResponse.avatar_url || '/default-avatar.png'
  };
}

// 页面中使用
import { normalizeUser } from './utils/data-transform.js';

const { users } = useDeps({}, {
  users: {
    source: '/api/users',
    transform: (res) => res.data.list.map(normalizeUser)
  }
});
```

#### 6.4 方案对比

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **transform 配置** | 单次转换 | 声明式、简洁 | 逻辑分散 |
| **订阅后处理** | 复杂逻辑 | 灵活 | 需手动管理状态 |
| **中间层封装** | 多处复用 | 统一、可测试 | 需维护工具库 |
| **后端适配** | 新项目 | 最简洁 | 需后端配合 |

**推荐**：优先用 `transform` 配置，复杂场景用中间层封装，新项目尽量推动后端适配（返回前端友好的格式）。


### 七、HTTP 客户端插件化

`useDeps()` 底层使用原生 `fetch`，但支持通过 `configureHttpClient()` 替换为任意 HTTP 客户端（如 Axios），保留拦截器、Token 注入等自定义逻辑。

#### 7.1 设计原则

- **零开销默认路径**：默认使用原生 `fetch`，函数引用直调，无额外查找
- **不强制依赖**：HTTP 客户端为可选插件，不引入则零体积增量
- **标准响应契约**：适配器返回 `{ ok, status, headers, json(), text() }` 统一格式
- **完全向后兼容**：现有 `useDeps()` / `useQuery()` 代码无需任何修改

#### 7.2 集成 Axios（示例）

```js
import { configureHttpClient } from 'kupola';
import { createAxiosAdapter } from 'kupola/adapters/axios';
import axios from 'axios';

// 创建 Axios 实例，配置拦截器
const api = axios.create({ baseURL: '/api', timeout: 10000 });

api.interceptors.request.use(config => {
  config.headers.Authorization = `Bearer ${getToken()}`;
  return config;
});

api.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) window.location.href = '/login';
    return Promise.reject(error);
  }
);

// 一行配置，之后所有 useDeps() 自动走 Axios
configureHttpClient(createAxiosAdapter(api));
```

#### 7.3 集成 @navios/http（示例）

```js
import { configureHttpClient } from 'kupola';
import { createNaviosAdapter } from 'kupola/adapters/navios-http';
import { create } from '@navios/http';

// 创建客户端，配置拦截器
const client = create({
  baseURL: '/api',
  headers: { 'X-Custom-Header': 'foobar' }
});

client.interceptors.request.use((config) => {
  config.headers['Authorization'] = `Bearer ${getToken()}`;
  return config;
});

client.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) window.location.href = '/login';
    return Promise.reject(error);
  }
);

// 一行配置，之后所有 useDeps() 自动走 @navios/http
configureHttpClient(createNaviosAdapter(client));
```

#### 7.4 标准响应接口

自定义 HTTP 客户端必须返回以下格式：

```js
{
  ok: boolean,                    // 请求是否成功（2xx）
  status: number,                 // HTTP 状态码
  headers: Record<string, string>, // 响应头
  json(): Promise<any>,           // 解析 JSON 数据
  text(): Promise<string>         // 解析文本数据
}
```

#### 7.5 自定义客户端

不依赖 Axios 时，可直接实现标准接口：

```js
import { configureHttpClient } from 'kupola';

configureHttpClient({
  fetch: async (url, options) => {
    const response = await myCustomFetch(url, options);
    return {
      ok: response.status < 400,
      status: response.status,
      headers: response.headers,
      json: () => Promise.resolve(response.parsedData),
      text: () => Promise.resolve(response.rawText)
    };
  }
});
```

#### 7.6 API 参考

| API | 说明 |
|-----|------|
| `configureHttpClient(client)` | 设置自定义 HTTP 客户端（必须提供 `fetch` 函数） |
| `getHttpClient()` | 获取当前 HTTP 客户端的 fetch 函数 |
| `resetHttpClient()` | 重置为原生 fetch（主要用于测试） |
| `createAxiosAdapter(instance)` | 从 Axios 实例创建适配器（`kupola/adapters/axios`） |
| `createNaviosAdapter(client)` | 从 @navios/http 实例创建适配器（`kupola/adapters/navios-http`） |


### 八、架构约束（不做的事）

| 不做 | 理由 |
|------|------|
| ❌ 完整 SPA 路由 | 非组件系统职责，已有 `depends.js` 处理数据 |
| ❌ 模板编译器 / `new Function` | 安全隐患，Phase 1 已删除 `mount.js` / `compiler.js` |
| ❌ Vue/React 兼容层 | Phase 1 已删除 adapters，避免"不如原版" |
| ❌ HTTP 客户端封装 | 用原生 `fetch` + 插件化适配器，不重复造轮子 |
| ❌ 测试工具 / Mock API | 不应在发布包中 |
