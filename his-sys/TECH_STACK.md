# HIS 系统技术栈文档

## 一、项目概述

医院信息管理系统（HIS）是一个基于前后端分离架构的医疗信息管理平台

## 二、技术栈

### 2.1 后端技术

| 技术 | 版本 | 说明 |
|------|------|------|
| Go | 1.22+ | 编程语言 |
| Gin | 1.9+ | Web 框架 |
| GORM | 1.25+ | ORM 框架 |
| PostgreSQL | 15+ | 数据库 |
| JWT | - | 认证授权 |

### 2.2 前端技术

| 技术 | 版本 | 说明 |
|------|------|------|
| Kupola | 1.3+ | 声明式 UI 组件系统 |

### 2.3 开发工具

| 工具 | 说明 |
|------|------|
| VS Code | IDE |
| Git | 版本控制 |

## 三、项目结构

```
his-sys/
├── his-app/                    # 后端 - Gin API 服务
│   ── main.go
├── his-web/                    # 前端 - Kupola 静态页面
│   └── index.html
├── kupola/                     # Kupola 缺陷记录与优化建议
│   ├── bugs.md                 # Bug 报告
│   ├── improvements.md         # 功能改进建议
│   └── performance.md          # 性能问题记录
└── TECH_STACK.md               # 技术栈文档（本文档）
```

## 四、Kupola 使用规范

### 4.1 核心原则

**重要规则**：如果在开发 his-sys 项目时使用 Kupola 发现存在缺陷或需要改进，**必须在 `kupola/` 文件夹下记录缺陷和 bug**，然后暂停 his-sys 系统的开发，总结如何优化修复 Kupola 并记录在专用文档中。

### 4.2 缺陷记录流程

1. 在 `kupola/` 对应文档中记录问题（Bug / 改进 / 性能）
2. 描述问题现象、复现步骤、期望行为
3. 暂停 his-sys 开发，回到 Kupola 主项目修复
4. 修复后在记录中标注状态为 `已修复`

## 五、开发环境

### 5.1 启动命令

```bash
# 启动后端服务
cd his-app
go run main.go

# 访问地址
http://localhost:8088/
```

### 5.2 端口配置

- 后端 API：8088
- 前端静态文件：通过后端服务访问

## 六、编码规范

### 6.1 后端规范

- 路由使用 RESTful 风格
- 控制器方法以 `Handle` 开头
- 错误处理使用统一响应格式
- 中间件按功能分类

### 6.2 前端规范

- 使用 Kupola 设计系统组件
- 组件通过 `data-*` 属性声明式初始化
- 样式遵循 Kupola CSS 命名约定（`ds-` 前缀）
