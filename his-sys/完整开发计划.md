# HIS 系统完整开发计划

## Task 1: 后端项目骨架搭建

创建 Go 后端项目基础结构：

```
his-app/
├── main.go                  # 入口，启动服务
├── config/
│   └── config.go            # 数据库配置、应用配置
├── database/
│   └── database.go          # GORM 连接 PostgreSQL
├── models/                  # 数据模型（后续 Task 填充）
├── handlers/                # 控制器（后续 Task 填充）
├── middleware/               # 中间件
│   ├── auth.go              # JWT 认证中间件
│   └── cors.go              # 跨域中间件
├── routes/
│   └── router.go            # 路由注册
└── go.mod / go.sum
```

关键配置：
- PostgreSQL 连接：`host=localhost user=postgres password=postgres dbname=his_sys port=5432`
- 服务端口：8088
- 静态文件服务：`/static/` 映射到 `his-web/` 目录

## Task 2: 数据库模型定义

根据 README 权限模型，创建以下 GORM 模型：

**`models/organization.go`** - 组织表
- `id`, `name`, `org_type`(group/branch), `parent_id`, `address`, `contact`, `status`, `created_at`, `updated_at`

**`models/department.go`** - 部门表
- `id`, `org_id`, `name`, `code`, `status`, `created_at`, `updated_at`

**`models/user.go`** - 用户表
- `id`, `username`, `password_hash`, `real_name`, `phone`, `email`, `org_id`, `org_type`, `status`, `last_login_at`, `created_at`, `updated_at`

**`models/role.go`** - 角色表
- `id`, `org_id`, `name`, `code`, `description`, `is_system`(预设角色标记), `status`, `created_at`

**`models/permission.go`** - 权限表
- `id`, `name`, `code`, `type`(menu/button/data), `parent_id`, `path`, `icon`, `sort_order`

**`models/role_permission.go`** - 角色权限关联
- `role_id`, `permission_id`

**`models/user_role.go`** - 用户角色关联
- `user_id`, `role_id`

## Task 3: 数据库迁移与种子数据

- `database/migrate.go` - AutoMigrate 所有模型
- `database/seed.go` - 初始化种子数据：
  - 创建默认集团组织
  - 创建系统预设角色（集团管理员、采购总监、分机构管理员、护士长、透析医师、库管员等）
  - 创建默认权限定义（菜单+操作）
  - 创建超级管理员账号

## Task 4: 认证模块

**`handlers/auth.go`**:
- `POST /api/auth/login` - 登录（返回 JWT token）
- `POST /api/auth/logout` - 登出
- `GET /api/auth/profile` - 获取当前用户信息（含角色、权限、菜单）
- `PUT /api/auth/password` - 修改密码

**`middleware/auth.go`**:
- JWT 解析中间件
- 从 token 提取 user_id, org_id, org_type 注入 context

## Task 5: 用户管理 CRUD

**`handlers/user.go`**:
- `GET /api/users` - 用户列表（分页、搜索、按组织/部门筛选）
- `POST /api/users` - 创建用户
- `GET /api/users/:id` - 用户详情
- `PUT /api/users/:id` - 更新用户
- `DELETE /api/users/:id` - 删除用户（软删除）
- `PUT /api/users/:id/status` - 启用/禁用用户
- `PUT /api/users/:id/roles` - 分配角色

## Task 6: 组织架构管理

**`handlers/organization.go`**:
- `GET /api/organizations` - 组织列表（树形结构）
- `POST /api/organizations` - 创建组织
- `GET /api/organizations/:id` - 组织详情
- `PUT /api/organizations/:id` - 更新组织
- `DELETE /api/organizations/:id` - 删除组织

**`handlers/department.go`**:
- `GET /api/departments` - 部门列表（按组织筛选）
- `POST /api/departments` - 创建部门
- `PUT /api/departments/:id` - 更新部门
- `DELETE /api/departments/:id` - 删除部门

## Task 7: 角色权限管理

**`handlers/role.go`**:
- `GET /api/roles` - 角色列表
- `POST /api/roles` - 创建角色
- `PUT /api/roles/:id` - 更新角色
- `DELETE /api/roles/:id` - 删除角色
- `PUT /api/roles/:id/permissions` - 分配权限

**`handlers/permission.go`**:
- `GET /api/permissions` - 权限树
- `GET /api/permissions/menus` - 获取当前用户菜单（根据角色动态计算）

## Task 8: 数据权限中间件

**`middleware/data_scope.go`**:
- 根据用户 org_type 和角色自动注入数据过滤条件
- 集团管理层：可查所有分机构数据（只读）
- 分机构用户：只能查本机构数据
- 通过 `GetDataScope(ctx)` 工具函数供各 handler 调用

## Task 9: 耗材管理基础模块

**`models/material.go`** - 耗材目录
- `id`, `name`, `code`, `spec`, `unit`, `category`, `manufacturer`, `registration_no`(注册证号), `status`

**`models/supplier.go`** - 供应商
- `id`, `name`, `contact`, `phone`, `address`, `license_no`, `status`

**`models/inventory.go`** - 库存
- `id`, `org_id`, `material_id`, `batch_no`, `quantity`, `expiry_date`, `location`

**`models/purchase_order.go`** - 采购订单
- `id`, `org_id`, `supplier_id`, `order_no`, `status`, `total_amount`, `created_by`

**`models/purchase_item.go`** - 采购明细
- `id`, `order_id`, `material_id`, `quantity`, `unit_price`, `amount`

**`handlers/material.go`** - 耗材目录 CRUD（集团独占增删改，分机构只读）
**`handlers/supplier.go`** - 供应商 CRUD（集团独占）
**`handlers/inventory.go`** - 库存查询（各机构只看自己的）
**`handlers/purchase.go`** - 采购订单管理

## Task 10: 前端登录页面

使用 Kupola 组件构建 `his-web/login.html`：
- 登录表单（用户名、密码、登录按钮）
- 表单验证
- 调用 `/api/auth/login` 接口
- Token 存储到 localStorage

## Task 11: 前端主布局与仪表盘

**`his-web/dashboard.html`** - 主布局：
- 左侧菜单（根据权限动态渲染）
- 顶部导航栏（用户信息、退出登录）
- 主内容区域

**`his-web/js/app.js`** - 前端核心逻辑：
- 路由管理（hash 路由）
- API 请求封装（带 JWT header）
- 菜单渲染
- 页面动态加载

## Task 12: 前端业务页面

- `his-web/pages/users.html` - 用户管理页面
- `his-web/pages/organizations.html` - 组织管理页面
- `his-web/pages/departments.html` - 部门管理页面
- `his-web/pages/roles.html` - 角色管理页面
- `his-web/pages/materials.html` - 耗材目录页面
- `his-web/pages/inventory.html` - 库存管理页面

## 开发顺序总结

按 Task 1 -> 12 顺序执行，每个 Task 完成后验证功能。最终产出：
- 完整的后端 RESTful API
- JWT 认证 + RBAC 权限控制
- 集团/分机构数据隔离
- Kupola 前端管理界面
